description: Audit recent changes for Mune privacy violations — unencrypted entry storage, RLS bypasses, third-party scripts on journal pages, plaintext in logs, and direct Claude API calls from client code
user-invocable: true
---

Check all recently modified files for Mune's architectural red lines:

1. **Unencrypted entries**: Find any Supabase insert/upsert on journal-related tables (journal_entries, entries, etc.) that doesn't first call `encrypt()` from `@mune/crypto`. Any path where plaintext entry content could reach a server is a violation.
2. **RLS bypass**: Look for missing `ENABLE ROW LEVEL SECURITY` in migrations, RLS policies that don't scope to `requesting_user_id()` (Mune uses the Clerk-JWT-text helper, NOT `auth.uid()` — the latter returns a uuid and silently fails against Clerk text IDs), or service role key usage in client-accessible code (`createBrowserClient` contexts). **FK-composition check:** when a user-scoped table has an FK to another user-scoped table (e.g. `chat_messages.session_id → chat_sessions.id`, `journal_entries.session_id → chat_sessions.id`), the INSERT and UPDATE `WITH CHECK` policy MUST also validate that the FK target belongs to `requesting_user_id()` via an `EXISTS (SELECT 1 FROM <parent> WHERE id = <child>.<fk> AND user_id = requesting_user_id())` clause. Validating only the child row's `user_id` lets an attacker INSERT a row pointing at another user's parent (integrity/abuse risk). Flag any INSERT/UPDATE policy on a child table that lacks this join.
3. **Third-party scripts on sensitive pages**: Check `app/(protected)/` layouts and any journal/reflection pages for `<Script>`, analytics tags, or external resource imports that would load on those routes.
4. **Plaintext in logs**: Search for `console.log`, `console.error`, `console.warn`, Sentry captures, or any logger calls in API routes or Edge Functions that could include entry text, reflection prompts, or user-identifiable content.
5. **AI layer violations**: All Claude API calls must route through the `/api/reflect` route handler. Flag any direct `anthropic.messages.create()`, `streamReflection()`, or `getAnthropicClient()` calls from client components or non-proxy routes.
6. **Pattern data exposure**: Check that no raw entry text — and no plaintext derived signal (sentiment, mood, word count, time-of-day, themes) — is stored as pattern/insight metadata. The metadata policy is empty-by-default: any populated `journal_entries.metadata` field without a written decision is a FAIL. Encrypted-blob metadata (ciphertext + iv) is the only acceptable form.
7. **Anthropic client centralization**: Every Anthropic API call must go through `getAnthropicClient()` in `packages/ai/src/client.ts`. Flag any `new Anthropic({...})` outside that file. Do NOT check for zero-data-retention headers — none exist in the Anthropic API (the `anthropic-no-log` header named in earlier versions of this skill was never real). Anthropic zero data retention is an organization-level retention configuration on the Anthropic account; it cannot be verified from the repo. If any doc or user-facing copy claims per-request ZDR headers, flag that claim as drift.
8. **Orphaned privacy-sensitive tables**: Some tables violate the privacy contract by their mere existence — `entry_patterns`, `mood_logs`, `sentiment_history`, `user_analytics`, or any table whose name suggests plaintext behavioral tracking. Check `supabase/migrations/` for any `CREATE TABLE` matching those names (or adjacent — `affect_log`, `emotion_history`, etc.) without a later `DROP TABLE` in a subsequent migration. Also grep `packages/db/src/queries/` for any query file referencing them. If a privacy-sensitive table exists in the live schema without a drop, flag as FAIL with: "Table `<name>` exists in schema but violates privacy contract. Add a DROP TABLE migration and remove all query files referencing it."

For each violation: show file path, line number, violation type, and a one-line fix. For each category with no issues: print "✓ clean".
